## Context

The Training tab currently has 6 metric-oriented sections (WoW card, volume chart, training load chart, run type breakdown, plan adherence, run timeline). The rewrite reorganizes these into 6 purpose-driven sections that answer runner questions, adds rolling 7-day windows to eliminate ISO week boundary artifacts, and shifts coaching staleness from sync-reactive to weekly cadence.

The change touches 4 layers: core analysis (rolling window), dashboard sections (cards + charts + template), CLI output, and MCP tools. The `weekly_agg` table and ISO week materialization remain intact — the rolling window is additive, not a replacement of the storage model.

Current codebase state: `compute_weekly_agg()` in analysis.py is the central aggregation function (~170 lines). It takes an ISO week string, queries activities for Mon-Sun, and returns a dict with all weekly metrics. ACWR and streak both read from `weekly_agg`. The dashboard reads from `weekly_agg` for all "this week" data. 24 call sites use ISO week logic across the codebase.

## Goals / Non-Goals

**Goals:**
- Training tab answers 4 runner questions without needing Claude or CLI: "how's my week?", "how did that run go?", "am I following the plan?", "what's the volume trend?"
- Rolling 7-day window eliminates Monday dead-dashboard and partial-week edge cases
- Coaching cadence shifts from sync-reactive to weekly ritual
- Per-run detail (splits, plan comparison, adaptation signals) surfaced in dashboard for the first time

**Non-Goals:**
- Migrating `weekly_agg` storage to a different schema — it stays ISO week, populated on sync
- Changing `compute_plan_adherence()` to rolling windows — plans are authored per calendar week
- Adding new tables or migrations — all data already exists
- Redesigning other tabs (Overview, Readiness, Fitness, Coach content) — only Coach staleness indicator changes
- Real-time dashboard — still static HTML generated by `fit report`

## Decisions

### 1. `compute_rolling_week()` as a thin wrapper over existing aggregation logic

**Decision**: Extract the core aggregation SQL from `compute_weekly_agg()` into a shared `_aggregate_date_range(conn, start_date, end_date)` internal function. Both `compute_weekly_agg()` (ISO week → Mon/Sun) and the new `compute_rolling_week(conn, end_date=None, window_days=7)` (today-6 → today) call it. The return dict has the same structure as a `weekly_agg` row.

**Why not just query activities directly in each consumer?** The aggregation logic is non-trivial (~80 lines of zone distribution, load sums, cadence averages, longest run, run type counts). Duplicating it in cards.py, alerts.py, cli.py, and mcp/server.py would be a maintenance hazard. One function, one truth.

**Why not replace `compute_weekly_agg()` entirely?** `weekly_agg` is still the right storage model for historical data. Charts showing 12+ weeks of volume read from pre-materialized rows — recomputing 12 rolling windows on every report generation would be slower and more complex. The rolling function is for "now" queries only.

**Alternative considered**: Query `weekly_agg` for the current ISO week and adjust. Rejected because the whole point is to not depend on ISO week boundaries — a Wednesday query should include last Thursday through Wednesday, which spans two ISO weeks.

### 2. ACWR computation: rolling for current, `weekly_agg` for chronic baseline

**Decision**: Current-week acute load comes from `compute_rolling_week()`. The 4-week chronic baseline still reads from `weekly_agg` (prior 4 ISO weeks). ACWR = rolling_acute / avg(chronic_from_weekly_agg).

**Why hybrid?** The chronic baseline is inherently historical — it's the average of 4 completed weeks. These are already materialized in `weekly_agg` and won't change. Only the acute (current) side needs the rolling window. This avoids recomputing 4 weeks of aggregation on every query.

**Trade-off**: There's a 0-6 day mismatch between the rolling acute window and the ISO-week chronic windows. This is acceptable because ACWR is a ratio indicator, not a precise measurement — the directional signal (overtraining vs. undertraining) is robust to small window alignment differences.

### 3. Volume Trend chart: `weekly_agg` for history + rolling for current bar

**Decision**: The Volume Trend chart reads completed ISO weeks from `weekly_agg` for all bars except the rightmost. The rightmost bar uses `compute_rolling_week()` for the current period. X-axis labels show the week end date (e.g., "Apr 12") instead of ISO week strings (e.g., "W15").

**Why not rolling bins for all bars?** 12 overlapping 7-day windows with 7-day stride are identical to ISO weeks shifted by day-of-week. The visual difference is negligible for historical bars, but the query is much simpler against `weekly_agg`. Only the current bar benefits from rolling.

**Alternative considered**: Fully rolling bins anchored to today. Rejected — adds complexity, doesn't change the story for historical data, and breaks the 1:1 mapping with `weekly_agg` rows that other parts of the system depend on.

### 4. Training tab data flow: cards.py owns all 6 sections

**Decision**: All new Training tab sections (`_this_week_card()`, `_training_objectives()`, `_last_7_days()`, `_weekly_plan_adherence()`) live in `cards.py`. The Volume Trend and Run Type Mix charts stay in `charts.py`. The template calls 4 card functions + 2 chart functions.

**Why cards.py, not a new module?** The existing module split is cards (HTML generation from data) vs charts (Chart.js config generation). The new sections are HTML cards, not charts. Adding a third module (`training.py`) would fragment the pattern without adding clarity. cards.py is already ~1300 lines — large but coherent. The new functions add ~400 lines.

**Alternative considered**: New `fit/report/sections/training.py` module. Rejected because it would require updating the import chain in engine.py and generator.py for a single-tab refactor. If cards.py exceeds ~2000 lines, split then.

### 5. Last 7 Days: server-side data, client-side expansion

**Decision**: All per-run data (header fields, split data, plan comparison, adaptation signals) is computed server-side in `_last_7_days()` and passed to the template as a list of dicts. The template renders all cards with detail sections hidden via CSS (`display:none`). A small JS toggle flips visibility on click — no AJAX, no lazy loading.

**Why not lazy-load detail on expand?** The dashboard is a static HTML file. There's no server to call. All data must be present at generation time. The cost is acceptable: 7 runs × ~2KB of detail HTML = ~14KB additional HTML. Split data for 7 runs is the heaviest part (~50 rows × 7 = 350 rows), but this is still small in a page that already inlines 3 JS libraries.

### 6. Adaptation signals: SQL query with self-join, not Python loop

**Decision**: The 4-week rolling comparison for each run queries activities once with a self-join: for each run in the last 7 days, get the avg pace and avg speed_per_bpm for the same `run_type` in the 28-day window before that run's date, with a HAVING count >= 2 filter. This returns one row per run with comparison data.

**Why SQL over Python?** The alternative is fetching all activities for 35 days and computing averages in Python. The SQL approach is a single query that the DB optimizer can handle efficiently. It also avoids loading all historical activities into Python memory.

**Trade-off**: The SQL is moderately complex (correlated subquery or CTE with window functions). But it's testable in isolation and the logic is well-defined.

### 7. Objectives row: 4 fixed slots, auto-derived only, no manual goals

**Decision**: The 4 canonical slots (Volume, Long Run, Z2, Consistency) are populated exclusively by `derive_objectives()` when a target race exists. All 4 activate together or all 4 are deactivated — no mixed state. Manual goals (`fit goal add`) do not feed into these slots. The slots map to derived objective names via prefix matching: "Peak volume" → Volume, "Long run" → Long Run, "Z2 compliance" → Z2, "Consistency" → Consistency.

**Why no manual goal mapping?** Manual goals have freeform names — there's no reliable way to map "Run 3x/week" to the Consistency slot without structured metadata. Adding that metadata (metric_source + threshold) is a separate change. For now, the binary state (target race → all active, no race → all deactivated) is simpler and eliminates an entire class of edge cases.

**Consequence**: The `fit goal add/list/complete` CLI group is removed. `fit target set/clear/show` is the sole path to objectives — setting a target race triggers `derive_objectives()` which populates all 4 slots. The `goals` table stays (it's where derived objectives are stored), but manual CRUD is deleted. The Overview tab's `_overview_objectives()` reads from the same auto-derived data.

**Consistency (streak) uses ISO weeks**: Streak counts consecutive ISO weeks with 3+ runs. It updates on Monday, not daily. This is intentional — rolling windows would cause counterintuitive drops when old runs age out of window boundaries on rest days. Streak, like plan adherence, measures commitment against calendar structure.

### 8. Coaching staleness: 7-day threshold, same code path

**Decision**: Change the staleness comparison in `_coaching()` (cards.py:348) from `report_date < last_sync` to `report_date < (today - 7 days)`. Same change in `check_dashboard_freshness()` (mcp/server.py). The 7-day threshold is hardcoded, not configurable — coaching cadence is a design decision, not a user preference.

**Why hardcoded?** Making it configurable suggests there's a valid reason to check coaching every day or every month. There isn't — weekly is the coaching cadence. Adding config adds complexity without value.

### 9. Alert partial-week suppression: delete, don't replace

**Decision**: Remove the `iso.weekday < 5` guard in alerts.py entirely. The rolling window means ACWR always reflects 7 days of data, so the guard's purpose (preventing false alerts from incomplete ISO weeks) no longer applies.

**Why not replace with a minimum-data guard?** The rolling window by definition always has 7 days. If there are 0 runs in 7 days, ACWR is 0 (undertraining) — which is a correct alert, not a false positive. The guard was solving an ISO-week-specific problem that rolling windows eliminate.

### 10. `generate_wow_sentence()`: parameter change, not rewrite

**Decision**: Change `generate_wow_sentence()` signature from `(current_week_data, prev_week_data)` to accept the same dict structure but sourced from `compute_rolling_week()` instead of `weekly_agg` ISO week rows. The function body (delta calculations, sentence templates) stays unchanged — it doesn't care where the data came from.

**Why not a new function?** The sentence logic is identical. Only the data source changes. Renaming or creating a new function would break existing callers for no benefit.

## Risks / Trade-offs

**[Rolling/ISO mismatch in ACWR]** → The acute load is rolling 7d, chronic is from ISO-week `weekly_agg` rows. This creates up to 6 days of window misalignment. Mitigation: ACWR is a directional indicator (safe/caution/danger bands), not a precise measurement. The mismatch doesn't change which band you're in for any realistic training pattern. If this proves problematic, a future change can compute chronic from rolling windows too.

**[cards.py growing large]** → Adding ~400 lines to an already ~1300-line module. Mitigation: The functions are self-contained (each produces one section's data). If the module exceeds ~2000 lines, extract training-specific functions to a `training.py` sub-module. The current change doesn't warrant this.

**[Split data in static HTML]** → Embedding per-km splits for 7 runs adds HTML weight. Mitigation: Split data is opt-in (requires `fit sync --splits`). Without splits, the detail section is lightweight (~200 bytes per run). With splits, ~14KB total — negligible in a page that inlines 200KB of JS.

**[Objective prefix matching is brittle]** → If `derive_objectives()` changes its naming convention, the 4-slot mapping breaks silently. Mitigation: Both functions are in our codebase. Add a test that verifies the mapping works for standard derived objectives.

**[Removing `fit goal` CLI]** → Deleting manual goal CRUD removes a user-facing feature. Mitigation: The feature is replaced, not just removed — `fit target set` + `derive_objectives()` is the new path. CLAUDE.md, MCP tool descriptions, and dashboard empty states must all be updated to reference `fit target set` instead of `fit goal add`.

**[`weekly_agg` and rolling window can diverge for "current week"]** → If `fit sync` hasn't run today, `weekly_agg` for the current ISO week may be stale while `compute_rolling_week()` shows fresh data from activities. Mitigation: This is the intended behavior — rolling window is always fresh (reads raw activities), `weekly_agg` is a cache updated on sync. Consumers that need "now" use rolling; consumers that need "history" use `weekly_agg`.

**[Coach tab stale threshold is hardcoded]** → No way for a user to adjust the 7-day coaching cadence. Mitigation: This is intentional. If user feedback shows demand for different cadences, make it configurable then. YAGNI for now.

## Resolved Questions

1. **Naming**: Hero card is "Last 7 Days" (not "This Week") — honest about the rolling window, consistent with the run detail section below it.

2. **4 canonical objectives**: Volume, Long Run, Z2 Compliance, Consistency — confirmed. Weight and pace are fitness outcomes (Fitness tab), not training inputs.

3. **Streak stays ISO-week based**: Rolling windows create a counterintuitive problem for streak — an old run aging out of a window boundary can shrink your streak on a rest day. Streak measures discipline against calendar structure ("I ran 3x every week for 12 weeks"). ISO weeks are the right model, same exception as plan adherence.

4. **Objective slots: auto-derived only, `fit goal` CLI removed**: The 4 slots show auto-derived targets when a target race exists, or all deactivated when no race. Binary state, no mixed. `fit goal add/list/complete` is removed — `fit target set` is the sole path to objectives. Deactivated prompt says "Set a target race with `fit target set`".
